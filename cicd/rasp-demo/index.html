内存马注入（OA 数据导入 → JVM 内存中的「秘密后门」）

通俗讲，内存马就是攻击者在 JVM 进程内存里留下的一扇「秘密后门」——后门逻辑本身不落盘、无文件，杀毒和文件扫描很难发现；攻击者通过隐藏地址传入命令，由 JVM 进程在内存中调用 exec 真实执行（如读取配置文件），结果回传。下方先说明正常应用如何通过 JVM 启动，以及黑客如何利用并注入内存马。

正常应用如何通过 JVM 启动 · 黑客如何利用 JVM 内存马

正常启动： OA 等 Java 应用以 jar/class 部署，启动时由操作系统拉起 JVM 进程（Java 虚拟机）。JVM 从磁盘加载 class 到内存，执行 main，启动 Web 容器并监听端口，所有业务逻辑运行在同一 JVM 进程的内存空间内；每次请求由该进程内的 Servlet/Controller 等处理。

黑客如何利用、如何知道能注入： 攻击者通过漏洞（如数据导入的反序列化、文件上传 RCE）在已运行的 JVM 进程中执行恶意代码。该代码在内存中动态注册 Filter、Servlet 或添加 URL 映射，不向磁盘写任何文件，因此传统杀软和文件监控难以发现。攻击者如何「知道」能注入：通过信息收集与漏洞利用——发现目标存在可触发的反序列化入口（如导入接口）、已知漏洞的依赖库等，构造恶意 payload 上传或提交，在服务端解析时触发 RCE，再在 RCE 中执行「向当前 JVM 注册隐藏后门」的代码，即完成内存马注入。

攻击路径演示（原理 + 技术表述）

步骤 0 · 攻击者如何访问到 OA

与命令注入相同：通过钓鱼、社工等拿到员工 OA 登录态，以该身份访问内网 OA。

步骤 1 · OA「数据导入」存在可被利用的漏洞

OA 的审批/Excel 导入等接口在解析上传文件时未严格校验，攻击者伪造「假文件」内藏恶意指令（技术上多为反序列化漏洞：服务器将数据还原成对象时触发恶意逻辑），解析时即获得在服务端执行任意代码的能力（RCE）。

步骤 2 · 在 JVM 内存中注入后门，无文件落地

攻击者利用 RCE 在当前 JVM 进程的内存中植入后门逻辑（如动态注册一个绑定到隐藏路径的 Servlet/Filter），不新建任何磁盘文件，故文件扫描无法发现；只要 OA 进程不重启，该后门常驻内存并可被访问。这类「仅存在于 JVM 内存、不落盘的后门」即内存马。

步骤 3 · 通过后门触发 exec，真实执行命令并窃取数据

攻击者向隐藏地址传入命令参数（如 cmd=cat /app/config/application.properties），内存马逻辑在 JVM 进程内调用 Runtime.exec(cmd)，由操作系统在容器内真实执行该命令（如启动子进程执行 cat 读取配置文件），并将标准输出回传。因此：内存马指「JVM 进程内存中的后门逻辑」；命令执行由该逻辑调 exec 完成，执行结果（如配置文件内容）即攻击者窃取到的数据。下方演示假设内存马已注入，模拟上述「传 cmd → JVM 调 exec 执行 → 回传结果」的过程。

SSRF 服务端请求伪造

通俗讲，SSRF就是「让服务器替攻击者发请求」——用户提交一个 URL，由服务端代为请求并返回内容；若接口未对 URL 做白名单或协议限制，攻击者可让服务器访问内网（如云元数据 http://169.254.169.254/）、file:///etc/passwd 等，从而窃取内网数据或绕过防火墙。模拟 OA「URL 预览」「资源抓取」场景。RASP 可钩子 URL.openConnection 等检测并阻断。

正常业务如何做 URL 预览 · 攻击者如何利用 SSRF

正常业务： OA 链接预览：用户填 URL，服务端做白名单或协议限制（如仅允许 https 且域名在白名单内），再代为请求取标题或内容。

攻击者如何利用： 若接口未校验用户传入的 url（如 /api/fetch-url?url=xxx），直接 URL.openStream() 等发起请求，攻击者传入内网地址（http://169.254.169.254/）、file:///etc/passwd 等，服务端以自身身份请求并回传结果，攻击者即可窃取内网数据或本地文件。下方演示假设接口存在 SSRF，模拟「填 URL → 服务端代为请求 → 回传结果」。
