# RASP 攻击演示（可扩展）

## 场景扩展与表达思路

- **双层表达**：每个场景的说明都尽量拆成两层——一层是**客户能直接听懂的业务语言**（这个功能在 OA 里是干嘛的、对谁有用），另一层在括号或补充段里给出**对应的技术术语/原理**（例如「命令注入」「反序列化 RCE」「JVM 内存马」等），避免只堆术语。
- **固定结构**：建议所有场景按同一结构编排，便于横向对比与后续扩展：
  - **入口/业务场景**：这个功能在 OA 中长什么样（菜单路径、页面名称）、正常是谁在用。
  - **攻击路径**：攻击者是如何进到 OA、如何发现这个入口、如何一步步利用（可拆成 0/1/2/3 步）。
  - **技术原理**：对应到哪些漏洞类型/关键 API，例如 `Runtime.exec`、反序列化、JVM 进程内存中的后门逻辑等。
- **配合简单图解**：在对外汇报 / PPT 中，建议每个场景至少配三类简易图：
  - 一张**架构/数据流示意图**（浏览器 → 网关/负载均衡 → OA → DB / 其他系统），标出攻击落点。
  - 一张**攻击路径时序图**（正常请求 vs 恶意请求，在哪一步被绕过/注入）。
  - 针对关键技术点的**小示意图**（例如「JVM 进程内存 + 内存马后门」「命令注入将用户输入拼进 shell」），帮助客户把术语和脑海中的图像建立对应关系。

后续新增场景时，尽量遵循以上表达风格：先让客户清楚「发生了什么、为什么危险」，再通过术语和图解补充「底层是怎么做到的」，既易懂又不失专业性。

用于**具体场景**演示多种攻击类型，当前包含**命令注入**、**内存马注入**、**XSS 跨站脚本**与**SSRF 服务端请求伪造**；页面按「攻击演示」分类编排，便于后续增加更多场景。

## 场景说明

- **业务含义**：模拟**甲方 OA 应用**（企业协同办公平台），整条攻击链强调为 **OA 系统自身的 API 漏洞** 导致；并回答客户常见疑问：「OA 是内网部署，黑客是怎么进到这个带漏洞的 API 的？」
- **攻击原因**：**OA 系统自身的 API 漏洞**（信息泄露、未做权限校验、命令注入均为 OA 接口设计/实现问题）。
- **漏洞场景**：采用 OA 常见的 **「发件服务器连接测试」**（邮件/通知设置中的 SMTP 测试），避免与主机安全、网络诊断类产品联想；接口为 <code>/api/smtp-test?address=xxx</code>。
- **攻击路径演示**（含「黑客如何接触到该 API」）：
  0. **攻击者如何能够访问到 OA**：通过钓鱼邮件、恶意链接、社工等获取员工 OA 登录态，从而在远端访问内网 OA 及各类 API。
  1. **OA 接口信息泄露**：`/api/internal-links` 向普通用户返回本不应暴露的内部链接，攻击者由此发现「发件服务器测试」地址。
  2. **OA 未做权限校验（越权）**：攻击者访问该地址时未做权限校验，可打开本应仅管理员可见的「发件服务器测试」页面。
  3. **OA 接口命令注入与数据窃取**：「发件服务器测试」API（`/api/smtp-test`）将用户输入的 SMTP 地址直接拼入 shell 执行，未做输入校验，攻击者注入命令并窃取数据。
- **漏洞点**：后端将 `address` 直接拼进 `sh -c "ping -c 2 " + address`；`/api/internal-links` 模拟 OA 接口信息泄露。
- **效果**：发件服务器测试页「攻击路径演示」默认展开，含步骤 0–3 及「执行注入演示」按钮。RASP 可挂到该 JVM 对 `Runtime.exec` 做钩子并告警/阻断。

### 内存马注入（OA 数据导入反序列化 → 无文件后门）

- **场景**：OA 常见的「审批数据导入」「Excel 模板导入」等接口存在**反序列化漏洞**，攻击者上传恶意序列化数据触发 RCE，在 JVM 内存中动态注册恶意 Filter/Servlet（内存马），**无文件落地**。随后通过隐藏路径传入 `cmd` 参数执行命令、窃取数据。
- **演示**：攻击演示 → 内存马注入。含攻击路径（步骤 0–3）与「内存马访问演示」：假设已注入，输入命令（如 `id`、`cat /etc/passwd`）调用 `/api/memory-shell?cmd=xxx`，回显结果；若为敏感输出则展示「安全事件说明」。
- **接口**：`GET /api/memory-shell?cmd=xxx`，后端将 `cmd` 交 shell 执行并返回输出，供 RASP 钩子检测。

### XSS 跨站脚本（反射型）

- **场景**：模拟 OA「站内搜索」「意见反馈」：用户输入被服务端**未经转义直接拼入 HTML 响应**（反射型 XSS）。攻击者提交恶意脚本，在受害者浏览器中执行，可窃取 Cookie、篡改页面。
- **演示**：攻击演示 → XSS 跨站脚本。输入关键词或恶意脚本（如 `<script>alert('XSS')</script>`），点击「搜索」；服务端返回的 HTML 在下方 iframe 中展示，脚本在 iframe 内执行。RASP 可对响应输出做钩子或特征检测。
- **接口**：`GET /api/xss-demo?q=xxx`，后端将 `q` 直接拼入 HTML 响应，未做转义。

### SSRF 服务端请求伪造

- **场景**：模拟 OA「URL 预览」「资源抓取」：用户提交 URL，**由服务端代为请求**并返回内容。接口未做 URL 白名单，攻击者可让服务器访问内网（如云元数据 `http://169.254.169.254/`）、`file:///etc/passwd` 等。
- **演示**：攻击演示 → SSRF 服务端请求伪造。输入 URL（如 `file:///etc/passwd` 或 `http://127.0.0.1:8080/`），点击「请求」；服务端发起请求并回显结果，敏感内容时展示「安全事件说明」。RASP 可钩子 `URL.openConnection` 等检测并阻断。
- **接口**：`GET /api/fetch-url?url=xxx`，后端按用户传入的 `url` 调用 `URL.openStream()` 等，未校验目标。

## 页面结构（可扩展）

- **主导航**：工作台 | **攻击演示** | 系统设置 | 帮助与反馈
- **攻击演示**：左侧为攻击类型列表（命令注入、内存马注入、XSS 跨站脚本、SSRF 服务端请求伪造、…），右侧为当前类型的说明与入口。
  - **命令注入**：说明 +「进入命令注入演示」→ 跳转到 OA 发件服务器测试场景（#/diagnostic）。
  - **内存马注入**：说明 + 攻击路径与「内存马访问演示」表单。
  - **XSS 跨站脚本**：说明 + 反射型 XSS 演示（搜索框 + iframe 展示服务端反射内容）。
  - **SSRF 服务端请求伪造**：说明 + URL 输入与「请求」按钮，展示服务端代为请求的结果。
- **新增攻击类型**：在 `index.html` 中 (1) 在 `.attack-sidebar` 内增加 `<a href="#/attack/新类型id" data-attack="新类型id">名称</a>`；(2) 增加 `<div id="attack-content-新类型id" class="attack-content">...</div>` 并填入说明与演示入口。如需新接口，在 `DiagnosticsServer.java` 中注册并实现对应 Handler。路由会按 id 自动展示对应内容。

## 启动

```bash
cd cicd/rasp-demo
docker compose up -d --build
```

若构建时拉取 JDK 镜像报 429，可改用单阶段构建（仅需 JRE）：先在本机执行 `./build.sh`（需已安装 JDK），再把 `docker-compose.yml` 里 `build.dockerfile` 改为 `Dockerfile.single`，然后重新 `docker compose up -d --build`。

访问：**http://\<主机\>:9093/**

- 输入 `127.0.0.1` 点「执行」→ 页面显示 ping 结果。
- 输入 `127.0.0.1 && whoami`（或 `id`）点「执行」→ 页面会显示 ping + 命令输出（如 `root` 或 `uid=...`）。

## RASP 验证步骤

1. **未挂 RASP**：上述注入请求正常返回，页面可见注入命令输出。
2. **挂上 RASP**：在容器或宿主机上以 `-javaagent:/path/to/rasp-agent.jar` 方式启动该 JVM（或通过 docker-compose 的 `JAVA_TOOL_OPTIONS` + volume 挂入 agent），再次发起注入请求。
3. **预期**：RASP 在 `Runtime.exec` 处检测到危险参数（或 shell 命令注入特征），产生告警/阻断；若为阻断，前端将收不到注入命令的输出或收到错误页。

## 与 VulnerableApp 的区别

| 对比项     | 本 demo (rasp-demo)     | VulnerableApp (java-app-test) |
|------------|-------------------------|--------------------------------|
| 用途       | 具体场景 + 前端回显 + RASP 联调 | 多漏洞、多 LEVEL 教学靶场     |
| 注入结果   | 直接回显到页面         | 多数关卡不回显                 |
| 界面       | 单页「诊断」表单       | 多关卡、说明文字多             |
| 代码       | 单文件，逻辑一目了然   | 大型项目                       |

本目录仅用于「在什么场景下做了命令注入、前端能显示、RASP 也能捕获」的端到端演示。
