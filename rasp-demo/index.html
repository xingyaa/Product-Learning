内存马注入（OA 数据导入 → JVM 内存中的「秘密后门」）

通俗讲，内存马就是攻击者在 JVM 进程内存里留下的一扇「秘密后门」——后门逻辑本身不落盘、无文件，杀毒和文件扫描很难发现；攻击者通过隐藏地址传入命令，由 JVM 进程在内存中调用 exec 真实执行（如读取配置文件），结果回传。下方先说明正常应用如何通过 JVM 启动，以及黑客如何利用并注入内存马。

正常应用如何通过 JVM 启动 · 黑客如何利用 JVM 内存马

正常启动： OA 等 Java 应用以 jar/class 部署，启动时由操作系统拉起 JVM 进程（Java 虚拟机）。JVM 从磁盘加载 class 到内存，执行 main，启动 Web 容器并监听端口，所有业务逻辑运行在同一 JVM 进程的内存空间内；每次请求由该进程内的 Servlet/Controller 等处理。

黑客如何利用、如何知道能注入： 攻击者通过漏洞（如数据导入的反序列化、文件上传 RCE）在已运行的 JVM 进程中执行恶意代码。该代码在内存中动态注册 Filter、Servlet 或添加 URL 映射，不向磁盘写任何文件，因此传统杀软和文件监控难以发现。攻击者如何「知道」能注入：通过信息收集与漏洞利用——发现目标存在可触发的反序列化入口（如导入接口）、已知漏洞的依赖库等，构造恶意 payload 上传或提交，在服务端解析时触发 RCE，再在 RCE 中执行「向当前 JVM 注册隐藏后门」的代码，即完成内存马注入。

攻击路径演示（原理 + 技术表述）

步骤 0 · 攻击者如何访问到 OA

与命令注入相同：通过钓鱼、社工等拿到员工 OA 登录态，以该身份访问内网 OA。

步骤 1 · OA「数据导入」存在可被利用的漏洞

OA 的审批/Excel 导入等接口在解析上传文件时未严格校验，攻击者伪造「假文件」内藏恶意指令（技术上多为反序列化漏洞：服务器将数据还原成对象时触发恶意逻辑），解析时即获得在服务端执行任意代码的能力（RCE）。

步骤 2 · 在 JVM 内存中注入后门，无文件落地

攻击者利用 RCE 在当前 JVM 进程的内存中植入后门逻辑（如动态注册一个绑定到隐藏路径的 Servlet/Filter），不新建任何磁盘文件，故文件扫描无法发现；只要 OA 进程不重启，该后门常驻内存并可被访问。这类「仅存在于 JVM 内存、不落盘的后门」即内存马。

步骤 3 · 通过后门触发 exec，真实执行命令并窃取数据

攻击者向隐藏地址传入命令参数（如 cmd=cat /app/config/application.properties），内存马逻辑在 JVM 进程内调用 Runtime.exec(cmd)，由操作系统在容器内真实执行该命令（如启动子进程执行 cat 读取配置文件），并将标准输出回传。因此：内存马指「JVM 进程内存中的后门逻辑」；命令执行由该逻辑调 exec 完成，执行结果（如配置文件内容）即攻击者窃取到的数据。下方演示假设内存马已注入，模拟上述「传 cmd → JVM 调 exec 执行 → 回传结果」的过程。

🔗 什么是 Gadget 链？

攻击者并没有自己发明零件，而是利用系统里已有的第三方库零件（如 Commons-Collections），按特定顺序串起来——每个零件单独看都合法，但串在一起在组装时就能触发 exec 执行命令。这个"特定顺序串起来的零件"就叫 gadget 链。

不需要管理员权限或二次操作
🛡️
传统防御难挡
每个 gadget 零件单独看都合法

反序列化攻击（OA 审批导入 → readObject 链路触发 RCE）

理解了上面的概念后再看攻击：攻击者把恶意指令「藏在」一个看似正常的数据包里（如 Excel/审批模板），服务端用 readObject() 「组装」时，恶意指令被自动执行——攻击者无需登录管理员，只要能调用「导入」接口就可远程执行命令（RCE），窃取数据库配置、密钥等。

🔗 一句话理解 JNI 劫持

Java 正常运行在「沙箱」里，做不了太危险的事。但 System.load 是沙箱上的一扇门——通过它加载的本地库（.so/.dll）直接在操作系统层面运行，不受 Java 安全限制。如果攻击者能控制「门后面加载哪个文件」，就相当于把自己的武器送进了沙箱内部。

在理解「XSS 攻击」之前，先搞清楚浏览器、HTML 和脚本之间的关系。

🌐 浏览器如何展示网页
用户访问 URL
服务端返回 HTML
浏览器解析并执行其中的脚本

用户输入: <script>alert(1)</script>

浏览器只显示文本，不执行
转义 = 把 < > 变成无害字符，脚本变纯文本

⚠️ 关键：不转义会怎样？

✅ 做了转义（安全）

vs
💣 没转义（XSS 漏洞）
用户输入脚本 浏览器执行了脚本！
攻击者的脚本在受害者浏览器里跑了

🔗 一句话理解 XSS

浏览器看到 HTML 里的 <script> 就会执行。正常情况下，服务端会把用户输入里的 < > 转成无害字符（转义），脚本就只是文字。但如果服务端没转义就直接写回 HTML，攻击者的脚本就会在受害者浏览器里执行——这就是 XSS，全称「跨站脚本攻击」。

🎭
篡改页面
伪造登录框骗取密码
👤
受害者的浏览器执行
攻击发生在客户端，服务端无感

💡 概念速览：什么是 SSRF（服务端请求伪造）？

在理解「SSRF 攻击」之前，先搞清楚「谁在发请求」这个关键问题。

👤 普通请求 — 用户自己发
用户在浏览器输入 URL
浏览器 → 目标网站
受防火墙限制，只能访问公网
用户自己发请求，受网络边界限制
🖥️ SSRF — 让服务器替你发
用户提交 URL 给服务端
服务端代为请求该 URL
服务器在内网，能访问内部资源！
服务器发请求，不受用户侧防火墙限制

⚠️ 关键：为什么「服务端代为请求」会出事？

✅ 有白名单校验（安全）
用户提交 URL 服务端校验：只允许公网
内网和本地文件请求被拒绝
vs
💣 没有校验（SSRF 漏洞）
攻击者传 file:///etc/passwd 服务器乖乖读取并返回！
攻击者借服务器的身份窃取数据

🔗 一句话理解 SSRF

用户自己访问不了内网，但服务器可以。SSRF 就是利用一个「让服务器代为请求」的功能，把服务器当跳板——让它去访问内网资源、读本地文件、甚至探测内部服务。攻击者在外面遥控，服务器在里面「跑腿」。

🏠
访问内网
借服务器身份探测/访问内部系统
📄
读取本地文件
file:// 协议读取 /etc/passwd 等
🔥
绕过防火墙
服务器在内网，不受外部边界限制

SSRF 服务端请求伪造

理解了上面的概念后再看攻击：用户提交 URL，由服务端代为请求并返回内容；若接口未做 URL 白名单，攻击者可让服务器访问内网或读取本地文件。模拟 OA「URL 预览」「资源抓取」场景。RASP 可钩子 URL.openConnection 等检测并阻断。
